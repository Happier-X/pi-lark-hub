# 技术设计

## 方案概览

将出站正文拆分责任集中到 `src/hub/feishu-outbound-format.ts`，由格式层先按 Markdown 组件安全上限切分正文，再按序构建卡片：

1. 短正文或可容纳的长正文：构建一张卡片，卡片内包含多个 `markdown` 元素。
2. 卡片 JSON 序列化后接近/超过飞书 30 KB 消息体上限：按已经切好的正文片段分组，构建多张连续卡片。
3. `NativeFeishuTransport.send` 按顺序发送分组卡片，全部成功后返回第一条真实 `message_id`，用于现有审批/回复绑定；中途失败直接抛错，不伪造成功绑定。
4. 卡片发送失败后，纯文本降级也按飞书文本限制分批发送，保证降级路径不再固定丢失正文。

## 数据流与边界

`Pi notify(title/body/event/actions)` → Hub `handleNotify` → `NativeFeishuTransport.send` → 出站格式层生成 `string[]` 卡片 content → 飞书 API 顺序发送 → 返回首条 message_id → 现有 binding/approval store。

协议字段、审批状态机和入站决策命令不变。多卡片只影响消息发送层；现有单条 message_id 绑定首条消息，审批仍由文本命令和 requestId 决策，不依赖新增回调字段。

## 切分规则

- 每个 Markdown 元素使用保守的 3000 字符正文上限，避免客户端/组件级限制；不再使用 3500 字符截断。
- 优先在换行处切分；若单行超过上限，再在字符边界切分。切分结果拼接后必须与规范化前正文完全一致，不插入“已截断”后缀。
- 尽可能让 fenced code block 成对落在同一片段；若代码块自身超过片段上限，允许内部切分，但测试保证不会丢字符。
- 将标题放在第一张卡片；后续卡片标题使用“标题（第 i/N 部分）”标识。单卡时保持原始标题。
- 卡片 JSON 必须按 UTF-8 字节数估算/校验，整体低于 30 KB，不能只按 JavaScript 字符数判断。
- 卡片内元素数量保持在飞书限制内；达到元素数量或字节上限就开始下一张卡片。

## 降级与失败

- 卡片发送失败时，按照文本消息的 150 KB 限制分批发送完整标题和正文；标题仅放在第一批，后续批次带“第 i/N 部分”。
- 卡片分批发送中途失败，抛出脱敏后的错误；不返回首条 id，避免将未完整发送的审批绑定为成功。
- 文本分批中途失败同样抛错；保留现有卡片失败原因 + 文本失败原因汇总。

## 兼容性与回滚

- `FeishuSendResult` 保持不变；调用方不需要知道内部批次。
- 短正文生成结构尽量保持现状（单个 markdown 元素、相同 header/template）。
- 回滚只需恢复格式层分批实现和 transport 顺序发送实现；协议和持久化格式无需迁移。
