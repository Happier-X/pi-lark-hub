# 实施计划

1. 更新出站格式层常量和类型，增加按换行/字符切分正文、按字节和元素数分组卡片、构建多卡片/多文本批次的纯函数。
2. 修改 `NativeFeishuTransport.send`，顺序发送所有卡片批次；失败后顺序发送完整文本批次；仍返回第一条成功消息的真实 `message_id`。
3. 为短正文、长正文、表格/代码块、审批/任务结束、多卡片和降级批次补充单元测试；验证拼接后的正文与输入一致。
4. 更新 `docs/lark-hub.md`、README 或 backend 出站合约，删除“单条截断”描述，记录分段和首条 message_id 绑定语义。
5. 运行 `npm run typecheck`、`npm test`、`git diff --check`，检查消息体字节限制与错误路径。

## 风险点与回滚点

- 风险：飞书组件真实限制可能小于公开消息体限制；通过 3000 字符元素上限和 30 KB 序列化字节预算留余量。
- 风险：多条消息部分成功；必须只在全部批次成功后返回绑定 id。
- 回滚点：格式层 API 和 transport 批量发送集中在两个文件，可独立恢复。
