# 修复飞书卡片内容被截断

## 目标

修复飞书出站卡片在内容较长时只展示正文前半段的问题，确保用户在飞书中能够完整看到需要决策的审批内容，而不是因出站层固定截断而丢失关键信息。

## 已确认事实

- 飞书出站路径位于 `src/hub/feishu-native.ts` 与 `src/hub/feishu-outbound-format.ts`。
- 当前 interactive 卡片只构建一个 `markdown` 元素。
- 当前 `CARD_MARKDOWN_MAX` 为 3500，超限正文会被截断并追加 `…（已截断）`。
- 当前纯文本降级也使用 3500 字符上限。
- 审批通知通过同一套出站 transport 发送，并携带 `title`、`body`、`event`、`requestId` 与可选 `actions`；审批状态机保存完整 body，但发送卡片时正文会先被截断。
- 项目现有合约明确规定“超长正文单条截断并追加后缀，不拆多条”，因此本任务若改为卡片内分段，需要同步调整该合约和测试。
- 飞书官方发送消息文档说明卡片消息请求体最大 30 KB；检索到的官方/相关资料显示 markdown 内容还存在组件级展示限制，因此不能仅把单个 markdown 上限继续调大。

## 需求

1. 所有出站卡片统一修复长正文展示，包括审批通知和任务结束通知；不得因本项目自身的固定截断逻辑静默丢失正文。
2. 方案应优先保持在同一条飞书卡片内展示，避免用户需要在多条消息之间拼接上下文。
3. 必须继续支持 Markdown 中的表格、代码块、列表和换行，并尽量避免在 Markdown 结构中间切断造成更严重的不可读内容。
4. 卡片发送失败时仍需保留纯文本降级能力，并明确降级路径的内容完整性边界。
5. 变更应有回归测试覆盖短正文、长正文、审批正文、任务结束正文及卡片/文本发送路径。

## 验收标准

- [ ] 长审批和长任务结束消息发送后，卡片中可读取完整正文；不再出现当前逻辑追加“已截断”导致信息缺失的情况。
- [ ] 卡片内容符合飞书卡片组件和消息体限制；超出单组件限制时采用可验证的分段/替代策略，而不是无提示丢弃正文。
- [ ] 现有短正文、标题、Markdown、发送失败降级和真实 `message_id` 绑定行为不回归。
- [ ] 相关单元测试、类型检查和 `git diff --check` 通过。
- [ ] 文档/规格中的出站卡片合约与实现一致。

## 范围外

- 不改变审批决策命令、审批状态机和 Pi 到 Hub 的协议字段。
- 不实现飞书历史消息编辑、文件附件或文档链接归档，除非调研证明卡片内分段无法满足完整展示且需要作为明确的备选方案。

## 已确认的产品决策

- 所有出站卡片统一采用长正文完整展示策略，不仅修复审批卡片。

## 已确认的产品决策（续）

- 当完整正文连同卡片结构超过飞书官方 30 KB 消息体硬限制时，拆成多条连续卡片，按“第 i/N 部分”标识，不保留单条截断方案。
